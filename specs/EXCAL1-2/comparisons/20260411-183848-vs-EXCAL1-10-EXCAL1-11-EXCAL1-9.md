## Implementation Choices Analysis

### [Spec] FR-001: App.tsx JSON parse error handling
| Ticket | Approach |
|--------|----------|
| EXCAL1-2 | Adds `console.error(e)` in catch |
| EXCAL1-10 | Adds `console.warn("Failed to parse postMessage event data as JSON", e)` |
| EXCAL1-11 | Adds type annotation `let data: any`, guards `typeof event.data === "string"`, handles object data, adds `console.error` with descriptive message, returns early on failure |
| EXCAL1-9 | Adds `console.error("Failed to parse JSON from postMessage event:", error)` with early `return` |

**Best**: EXCAL1-11 — Most robust: type-guards input, handles both string and object message data, descriptive error message.

### [Spec] FR-002: clipboard.ts JSON parse error handling
| Ticket | Approach |
|--------|----------|
| EXCAL1-2 | Adds `console.error(e)` |
| EXCAL1-10 | Adds `console.warn("Failed to parse clipboard data as Excalidraw JSON", e)` |
| EXCAL1-11 | Adds early guard `if (!serializedClipboardData.startsWith("{"))` to avoid unnecessary parse attempts; catch has comment only |
| EXCAL1-9 | Adds `console.error("Failed to parse clipboard data:", error)` |

**Best**: EXCAL1-11 — Preventive approach avoids parsing non-JSON data entirely, though the catch block should still log.

### [Spec] FR-003: CustomStats.tsx clipboard copy error handling
| Ticket | Approach |
|--------|----------|
| EXCAL1-2 | Extracts `handleVersionCopy` function, adds `console.error`, adds `useMemo` for version, refactors JSX |
| EXCAL1-10 | Adds `console.error("Failed to copy version to clipboard", e)` in-place |
| EXCAL1-11 | Adds `props.setToast(t("errors.copyToSystemClipboardFailed"))` — user-facing feedback with i18n |
| EXCAL1-9 | Adds `console.error("Failed to copy version to clipboard:", error)` in-place |

**Best**: EXCAL1-11 — User-facing error feedback is more valuable than silent logging (spec edge case noted: "critical path → user-visible notification").

### [Spec] FR-005: utils.ts localStorage parse error handling
| Ticket | Approach |
|--------|----------|
| EXCAL1-2 | Adds `console.error(e)` |
| EXCAL1-10 | Adds `console.warn("Failed to parse feature flags from localStorage", e)` |
| EXCAL1-11 | Assigns `DEFAULT_FEATURE_FLAGS` before try block (safety), adds `console.error("unable to parse feature flags", error)`, removes redundant fallback |
| EXCAL1-9 | Adds `console.error("Failed to parse feature flags from localStorage:", error)` |

**Best**: EXCAL1-11 — Eliminates null-safety risk by assigning defaults before try, simplifies return expression.

### Decision Point 1: Scope Discipline
| Ticket | Approach | Trade-offs |
|--------|----------|------------|
| EXCAL1-2 | Extensive refactoring: useMemo, animation rewrite, memoize simplification, polygon restructuring, comments | High risk of regressions in unrelated code |
| EXCAL1-10 | Minimal changes, only catch blocks | Focused, low risk |
| EXCAL1-11 | Targeted improvements beyond catch blocks (guards, types, tests) | Moderate scope expansion, but all related to error handling |
| EXCAL1-9 | Minimal changes, only catch blocks + one early return | Focused, low risk |

**Best**: EXCAL1-9/EXCAL1-10 — Most disciplined scope. EXCAL1-11 is acceptable as its expansions are error-handling-related.
