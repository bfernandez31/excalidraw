## Implementation Choices Analysis

### [Spec] Clipboard copy failure surfacing (CustomStats.tsx)
| Ticket | Approach |
|--------|----------|
| EXCAL1-11 | Shows user-facing toast via `props.setToast(t("errors.copyToSystemClipboardFailed"))` |
| EXCAL1-10 | `console.error` only — user gets no feedback |
| EXCAL1-9 | `console.error` only — user gets no feedback |

**Best**: EXCAL1-11 — Spec explicitly requires surfacing clipboard copy failure to the user. Only EXCAL1-11 delivers this.

### [Spec] Clipboard JSON parsing (keep non-JSON quiet)
| Ticket | Approach |
|--------|----------|
| EXCAL1-11 | Early guard `!trimStart().startsWith("{")` skips JSON.parse for plaintext; catch has comment only |
| EXCAL1-10 | `console.warn` in catch — logs every plaintext paste as a warning |
| EXCAL1-9 | `console.error` in catch — logs every plaintext paste as an error |

**Best**: EXCAL1-11 — Correctly distinguishes between malformed JSON (object-like but invalid) and normal plaintext. The early guard avoids noisy logging for legitimate non-JSON clipboard content, exactly as the spec requests.

### [Spec] Test coverage for fixes
| Ticket | Approach |
|--------|----------|
| EXCAL1-11 | 5 new test files: CustomStats.test.tsx, utils.test.ts, elementLink.test.ts, clipboard.test.ts, App.test.tsx |
| EXCAL1-10 | No tests |
| EXCAL1-9 | No tests |

**Best**: EXCAL1-11 — Spec requires "validate with impacted tests only." EXCAL1-11 adds targeted tests for each fix site.

### Decision Point 1: Feature flag fallback safety (utils.ts)
| Ticket | Approach | Trade-offs |
|--------|----------|------------|
| EXCAL1-11 | Sets `featureFlags = DEFAULT_FEATURE_FLAGS` before try; parse overwrites on success. Removes redundant `featureFlags \|\| DEFAULT_FEATURE_FLAGS` in return. | Fixes a latent bug: original code left `featureFlags` as `undefined` on parse failure |
| EXCAL1-10 | Adds `console.warn` to catch, no structural change | Leaves the `featureFlags \|\| DEFAULT_FEATURE_FLAGS` fallback pattern intact |
| EXCAL1-9 | Adds `console.error` to catch, no structural change | Same as EXCAL1-10 |

**Best**: EXCAL1-11 — Proactively fixes the initialization order so the fallback is guaranteed, not reliant on a runtime `||` guard.

### Decision Point 2: postMessage handling (App.tsx)
| Ticket | Approach | Trade-offs |
|--------|----------|------------|
| EXCAL1-11 | Type-checks `event.data`, handles string (JSON.parse) and object cases separately, early return on parse failure | More robust: handles non-string postMessage data without crashing |
| EXCAL1-10 | Adds `console.warn` in catch | Minimal change, still attempts JSON.parse on non-string data |
| EXCAL1-9 | Adds `console.error` and early return in catch | Better than EXCAL1-10 (early return) but still attempts JSON.parse on non-string data |

**Best**: EXCAL1-11 — Correctly handles the case where `event.data` is already an object (some embeddable frames send structured data).

### Decision Point 3: Log level selection
| Ticket | Approach | Trade-offs |
|--------|----------|------------|
| EXCAL1-11 | `console.error` for truly exceptional failures; comment-only for expected non-JSON | Appropriate severity levels |
| EXCAL1-10 | `console.warn` everywhere | Consistent but incorrect — clipboard copy failure is an error, not a warning |
| EXCAL1-9 | `console.error` everywhere | Over-logs normal flows (plaintext paste logged as error) |

**Best**: EXCAL1-11 — Contextual log levels match the severity of each case.

### Decision Point 4: elementLink.ts simplification
| Ticket | Approach | Trade-offs |
|--------|----------|------------|
| EXCAL1-11 | Simplified: `return new URL(url).searchParams.get(ELEMENT_LINK_KEY)` — single expression | Cleaner; `.get()` returns `null` when key is absent, making the `has()` check redundant |
| EXCAL1-10 | Same simplification | Same benefit |
| EXCAL1-9 | Kept original verbose pattern with `has()` + `get()` | No simplification |

**Best**: EXCAL1-11 / EXCAL1-10 (tie) — Both correctly recognize `.get()` returns `null` when absent.
